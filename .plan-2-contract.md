# Plan 2 Shared Ground-Truth Contract

This document defines the SHARED API surface, naming, and codebase facts that all four Plan 2 chunks (2a/b/c/d) must agree on. Every plan-fix subagent uses this as the source of truth when resolving cross-plan inconsistencies and codebase-mismatch BLOCKERs.

## Dispatcher API (defined by Plan 2a, used by 2b/c/d)

Module: `server/utils/scheduledMessageDispatcher.js`

```js
function registerHandler(messageType, fn, options = {})
// options: {
//   offsetFromEventDate: number | null  // seconds. Negative for pre-event, positive for post-event, null for non-event-anchored (e.g. drip touches)
//   anchor: 'event_date' | 'balance_due_date' | 'created_at' | 'completed_at' | 'now'  // default 'event_date'
//   category: 'operational' | 'marketing'  // default 'operational'
// }

function getHandlerMeta(messageType): { offsetFromEventDate, anchor, category } | null

async function dispatchPending()  // the scheduler-tick function (NOT processScheduledMessages)
```

**Handler registration timing**: handlers register at module-load time via top-level `registerHandler(...)` calls inside `scheduledMessageDispatcher.js` itself. There is NO separate `moneyPathHandlers.js` or `moneyPathHandlers.registerAll()`. Plans 2c/2d's handler modules (`preEventHandlers.js`, `marketingHandlers.js`) DO export `registerAll()` because they're separate files, but Plan 2a's handlers register directly.

**Handler invocation signature**:
```js
async (handlerCtx) => void  // throws to mark row failed
// handlerCtx: { entity, recipient, scheduledMessage }
// entity: full proposal/shift/etc row, accessed via entity.id (NOT entity.entity_id)
// recipient: full clients/users row, accessed via recipient.id (NOT recipient.recipient_id)
// scheduledMessage: the scheduled_messages row itself
```

## Suppression helper (defined by Plan 2a, used by 2b/c)

Module: `server/utils/messageSuppression.js`

```js
async function shouldSendImmediate({ proposal, client, channel }):
  { ok: true } | { ok: false, reason: 'archived' | 'channel_disabled' | 'bad_contact' }
```

Plans 2b and 2c MUST call this before every immediate sendEmail. Plan 2d's review request handler is dispatcher-driven (uses dispatcher's built-in suppression) so does NOT call shouldSendImmediate.

## Scheduling helper (defined by Plan 2a)

Module: `server/utils/messageScheduling.js`

```js
async function scheduleMessage({ entityType, entityId, messageType, recipientType, recipientId, channel, scheduledFor }):
  { id: number, status: 'inserted' | 'duplicate' }
```

**Idempotency**: uses partial unique index on `(entity_id, entity_type, message_type, recipient_id, recipient_type, channel) WHERE status = 'pending'`. The `ON CONFLICT` clause MUST use the column-list inference form, NOT `ON CONSTRAINT name` (PostgreSQL does not accept named unique INDEXES — only named CONSTRAINTS — in `ON CONFLICT ON CONSTRAINT`).

Correct:
```sql
ON CONFLICT (entity_id, entity_type, message_type, recipient_id, recipient_type, channel)
WHERE status = 'pending'
DO NOTHING
```

## Codebase facts (don't get these wrong)

- **Stripe webhook file**: `server/routes/stripe.js` (NOT `server/routes/stripeWebhook.js`)
- **Webhook call order in current code**: `sendPaymentNotifications(proposalId, ...)` runs BEFORE `createEventShifts(proposalId)` in the post-commit notifier. The drink plan does NOT exist when sendPaymentNotifications fires. Plan 2b MUST swap this order so orientation can read the drink_plan_token.
- **proposals.client_name / proposals.client_email do not exist.** Client name and email live on the `clients` table, linked via `proposals.client_id`. Joins: `SELECT p.*, c.name AS client_name, c.email AS client_email FROM proposals p JOIN clients c ON c.id = p.client_id`.
- **proposals.token is UUID type with default `gen_random_uuid()`.** Test fixtures should either omit it (default fires) or explicitly use `gen_random_uuid()` — never a string literal.
- **proposals.status enum**: `draft, sent, viewed, modified, accepted, deposit_paid, balance_paid, confirmed, completed, archived`. No `cancelled` (migrated to archived in Plan 1).
- **shift_assignments table does NOT exist.** Shifts-to-staff linkage uses `shift_requests` (status: pending/approved/denied) or `shifts.assigned_user_id` (single bartender). Verify the actual schema before writing queries.
- **payment_profiles columns**: `venmo_handle`, `cashapp_handle`, `paypal_url`. NO `zelle_handle`. Either add a migration to introduce it or drop Zelle from the tip-handle flow.
- **event_start_time is wall-clock event-local time stored as a string like '18:00' or '6:00 PM'**, NOT a UTC ISO timestamp. Do NOT parse it as UTC via `new Date('2026-06-15T18:00:00Z')`. Use a string-split approach or pair with `event_timezone` to construct the right UTC moment.
- **shouldSendImmediate requires `client.communication_preferences`, `client.email_status`, `client.phone_status`**. SELECTs that pass `client` to shouldSendImmediate MUST include these columns (often via `JOIN clients c ON c.id = p.client_id`).
- **email_sequence_enrollments engine coexistence**: the existing "Abandoned Quote Followup" campaign (schema.sql:1561-1600) drives a separate drip via `emailSequenceScheduler.js`. Plan 2d's new drip via `scheduled_messages` MUST decommission or carve out non-overlapping triggers — otherwise double-sends. Resolution: Plan 2d Task 0 pauses the existing campaign via `UPDATE email_campaigns SET status = 'paused' WHERE name = 'Abandoned Quote Followup' AND type = 'sequence' AND status <> 'paused'`. Engine boundary going forward: old engine = raw lead nurture (`email_leads` table) only; new dispatcher = proposal-anchored touches.
- **Computed-anchor cascade hook (Plan 2c ↔ 2d)**: `new_year_hello`'s scheduled_for is computed via calendar math (Jan 2 of event_year at 10am event-local), not a fixed offset, so it registers with `{ anchor: 'event_date', offsetFromEventDate: null }`. Plan 2c's `reanchorPendingMessages` cascade MUST detect this metadata combination and dispatch to a per-type recompute helper instead of leaving the row alone. Plan 2d exports `recomputeNewYearHelloForProposal(proposalId)` (DELETE + re-evaluate eligibility). For every pending `new_year_hello` row touched by the cascade, Plan 2c must `await recomputeNewYearHelloForProposal(proposalId)`. Generalizing: any message_type with `{ anchor: 'event_date', offsetFromEventDate: null }` should register a sibling `recompute<MessageType>ForProposal` helper exported by the owning plan, and Plan 2c's cascade should look it up by convention.
- **Drip suppression on sign+pay (Plan 2d Task 9 Step 3)**: the Stripe webhook post-commit notifier must suppress pending `drip_touch_2` / `drip_touch_4` / `drip_touch_5_email` rows for the proposal when status moves to `deposit_paid`. Plan 2d's existing `cancelMarketingForProposal` only fires on archive, so this is an additional sibling try/catch in the sign+pay hook (specifically in `server/routes/stripe.js`).
- **emailTemplates.js is 756 lines today; 1000-line hard cap via .husky/check-file-size.sh.** Plans 2a/2b/2c combined will breach the cap. Plan 2b (largest contributor) should include a task to split into `lifecycleEmailTemplates.js`. Plan 2d already correctly carves out `marketingEmailTemplates.js`.
