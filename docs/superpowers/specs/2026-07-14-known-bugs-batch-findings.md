# B10-strand-heal-renotify — STALE_DIAGNOSIS_BUT_BUG_REAL (sensitive=true)

## root_cause
The thumbtack duplicate-heal (fix-perim, 7b01a78) uses thumbtack_leads.proposal_id as its "post-commit steps never ran" marker, but that marker (a) only becomes true when the DRAFT sub-step succeeds, in a separate transaction that commits seconds after the lead row's COMMIT, and (b) says nothing about the NOTIFY sub-step, which runs unconditionally and is never claimed or recorded. The heal predicate therefore cannot distinguish "original attempt hard-crashed after COMMIT" (heal correct, notify never happened) from "original attempt is still alive in its post-commit tail" (heal wrong, notify is about to happen). A provider retry landing in the lead-COMMIT→draft-COMMIT window (or any later duplicate after a transient draft failure) satisfies `client_id && !proposal_id` and re-runs notifyAdminCategory concurrently with / after the original's notify. Primary data is safe (negotiation_id UNIQUE at schema.sql:1732; draft serialized by SELECT..FOR UPDATE + proposal_id guard at thumbtackProposalDraft.js:170-176); only the admin email double-fires. The calcom half of the recorded diagnosis was never mechanically possible at HEAD: its handlers send no notifications, so its heal/retry overlap is notification-inert.

## fix_design
Touch ONLY server/routes/thumbtack.js (no calcom change, no schema change, no claim table — single-instance, minimal). Treat thumbtack_leads.created_at as the original attempt's implicit lease: the original request starts its post-commit tail immediately after the COMMIT that stamps created_at, so a duplicate arriving within a short window must presume the tail is still in flight. Concretely: (1) extend the dedupe SELECT (thumbtack.js:375-377) to `SELECT id, client_id, proposal_id, (created_at < NOW() - INTERVAL '10 minutes') AS heal_eligible FROM thumbtack_leads WHERE negotiation_id = $1` (DB clock, no app/DB skew); (2) in the duplicate branch, split the heal predicate: `if (row.client_id && !row.proposal_id) { if (!row.heal_eligible) return res.status(503).json({ status: 'retry_later' }); ...existing heal unchanged... }`. The 503 (not 200 'duplicate') is load-bearing: it keeps the provider's retry chain alive, so a genuine crash-after-commit strand whose first retry arrives inside the window is NOT permanently lost — Thumbtack retries with backoff, and once the window passes (and proposal_id is still NULL) the heal fires; meanwhile in the overlap case the original finishes within seconds, sets proposal_id, and the next retry short-circuits as a plain 200 'duplicate'. This does not touch the F1/F1b calcom mechanism at all, adds no multi-instance machinery, and honors rare-duplicate-over-lost: the only residual duplicates are (i) an original request still alive >10 min into its post-commit tail overlapping a >10-min-late duplicate (vanishingly rare) and (ii) one extra admin email when a heal repairs a transiently-failed draft whose original notify succeeded — the accepted price of healing the missing draft. Alternatives considered and rejected: notify-after-settle reordering (notify already IS the last step; the problem is two executors, not ordering); a notified_at mark-after-send column (schema change, still racy — both executors check before either sends); claim-before-send UPDATE..RETURNING on a notified_at column (closes the race but converts a send failure into silent loss — violates the recorded F1b duplicate-over-lost preference). Genuinely open knobs for Dallas: window length (10 min recommended — generous vs any plausible request lifetime, tiny vs Thumbtack's retry horizon) and 503-vs-200 on the gated fresh strand (503 recommended; 200 would end the provider retry chain and narrow heal coverage to >10-min duplicates only). Do NOT change server/routes/calcom.js: no bug exists there at HEAD, it is a hardened sensitive webhook (protect-working-paths), and any gate there would have to 503 instead of 200 or it re-opens F1 strand-loss — record in the spec that IF consult notifications are ever added to calcom handlers, the same received_at-age + non-2xx gate must be added to its strand-heal at that time. Small enabling refactor in thumbtack.js: route notifyAdminCategory through the existing `_deps`/__setDeps test seam (:14-15) so tests can count notify calls; zero prod behavior change.

# B11-voice-deadleg-dedup — CONFIRMED (sensitive=true)

## root_cause
Non-atomic check-then-act (TOCTOU) on the dead-leg alert dedup. The dedup marker (the call_audit row) is deliberately written only AFTER a successful Telegram send (to preserve retry-on-failed-send), which places the awaited `sendTelegramMessage` HTTP call directly between the existence-check and the row-insert. Two Twilio status callbacks for the same (CallSid, dead-status) — at-least-once redelivery and/or concurrent leg callbacks — interleave inside that await window: both see no row, both send, both insert. Single prod instance does not help because Node's event loop still yields at every await; the network round-trip to the Telegram Bot API is a large, reliable interleaving window. Result: Zul gets a duplicate "that call didn't connect" Telegram alert (and a duplicate audit row).

## current_evidence
The dead-leg dedup at HEAD is check-then-act (TOCTOU), unchanged in mechanism since commit 7b01a78 (2026-07-03, lane fix-perim) added it; voice.js/pendingCall.js untouched since.

server/routes/voice.js:58-65 — `auditRowExists(callSid, status)` is a plain SELECT probe: `SELECT 1 FROM call_audit WHERE call_sid = $1 AND status = $2 LIMIT 1`.

server/routes/voice.js:164-210 — `/status` handler. The dead-leg branch (line 168 `if (DEAD_STATUSES.has(status))`) runs:
  - line 175: `alreadyAudited = await _deps.auditRowExists(callSid, status)` (the CHECK)
  - line 179-182: `if (alreadyAudited) { res.status(204).end(); return; }`
  - line 185-192: `await _deps.sendTelegramMessage(...)` (the SEND — a wide awaited network window)
  - line 196-207: `if (alertDelivered) { await _deps.recordAudit({...callSid, status}) }` (the INSERT, deferred until AFTER the send)
The CHECK (175) and the INSERT (198) are separated by the awaited Telegram HTTP send (187). Node interleaves at every `await`, so two concurrent status callbacks for the same (CallSid, dead-status) BOTH pass `auditRowExists` (neither has inserted yet), BOTH send the Telegram alert, BOTH append a call_audit row.

server/utils/pendingCall.js:82-88 — `recordAudit` is a bare `INSERT INTO call_audit (...) VALUES (...)`, no ON CONFLICT / no atomic guard.

server/db/schema.sql:3502-3513 — `call_audit` table has ONLY `idx_call_audit_created_at`; grep for UNIQUE on call_audit returns nothing. No uniqueness on (call_sid, status), so nothing enforces single-alert at the DB layer.

The 2026-07-03 dedup (comment voice.js:169-172, 193-195) only closes the SEQUENTIAL case where a retry arrives after the first request already committed its audit row; it does not close the concurrent window. Zero money/data impact — call_audit dead-leg rows are forensic only; the daily spend cap (`countPlacedSince`, pendingCall.js:69-78) counts `status='placed'` and is unaffected.

## fix_design
Make the "claim the right to alert" step atomic by moving the durable write BEFORE the send and using an INSERT ... ON CONFLICT DO NOTHING claim, while preserving the existing retry-on-failed-send property via a release-on-failure DELETE. Three source edits + tests.

1) server/db/schema.sql (SENSITIVE, idempotent) — add a PARTIAL unique index inside the VA-calling block, immediately after `idx_call_audit_created_at` (~line 3513) so `schema.vaCalling.test.js`'s `vaCallingDdl()` slice (from the `-- Zul VA Calling` marker at line 3467 to EOF) picks it up:
   CREATE UNIQUE INDEX IF NOT EXISTS uq_call_audit_dead_leg
     ON call_audit (call_sid, status)
     WHERE call_sid IS NOT NULL
       AND status IN ('no-answer','busy','failed','canceled');
   PARTIAL (not a full UNIQUE) on purpose: call_audit also holds callSid=NULL rows (`rejected_cap`/`rejected_validation` and the placement-`failed` at telegram.js:168) plus one `placed` row per call; scoping to non-null call_sid + the four dead statuses constrains ONLY the dead-leg callback rows we dedup, so it cannot silently DO-NOTHING a spend-cap-relevant insert. (call_sid,'placed') and (call_sid,'failed') are distinct keys, so a call getting both a placed and a dead-leg row is unaffected.

2) server/utils/pendingCall.js — add two helpers and export them:
   claimDeadLegAudit({ triggeredBy, callSid, status }) →
     INSERT INTO call_audit (triggered_by, target_e164, call_sid, status)
     VALUES ($1, NULL, $2, $3)
     ON CONFLICT (call_sid, status)
       WHERE call_sid IS NOT NULL AND status IN ('no-answer','busy','failed','canceled')
       DO NOTHING
     RETURNING id;
     return rows.length > 0;  // true = this caller won the claim and owns the alert
   releaseDeadLegAudit({ callSid, status }) →
     DELETE FROM call_audit WHERE call_sid = $1 AND status = $2;
   (The `ON CONFLICT (cols) WHERE <index_predicate>` form is valid partial-index arbiter inference in Postgres.)

3) server/routes/voice.js — rewire the dead-leg branch of `/status` (replace lines ~173-207). Read `allowed` once at the top of the branch; compute `triggeredBy = allowed ? Number(allowed) : null`. Then:
   - Try `claimed = await _deps.claimDeadLegAudit({ triggeredBy, callSid, status })`. On a thrown probe error, log and set `claimed = true` (best-effort fall-through: a rare double-ping beats a swallowed alert — preserves the existing HEAD intent; no durable row exists in this DB-error path so no release is attempted).
   - `if (!claimed) { res.status(204).end(); return; }`  (someone else owns this exact (CallSid,status)).
   - We own it. If `allowed`, `await _deps.sendTelegramMessage(...)`; on send failure, log AND `await _deps.releaseDeadLegAudit({ callSid, status })` so Twilio's next at-least-once redelivery can re-claim and re-send (this preserves the deliberate 2026-07-03 "resend on failed send" property that pure claim-first would regress).
   - `res.status(204).end()`.
   Update the DI seam: add `claimDeadLegAudit`/`releaseDeadLegAudit` to the import and to `_deps` (line 70); drop `auditRowExists` and `recordAudit` from voice.js (no longer referenced there — `recordAudit` stays exported for telegram.js).

Why this shape (why not the two simpler alternatives): (a) pure claim-first (write row before send, no release) closes the race but REGRESSES the failed-send case — a transient Telegram failure leaves a durable row that suppresses the alert for the 30-day retention, so Zul never learns; the release-on-failure DELETE restores retry-on-failure at a 3-line cost. (b) An UPDATE ... WHERE alerted_at IS NULL RETURNING guard on the pending_call row was considered and rejected: pending_call rows are pruned/replaced (call_sid reset to NULL on a new target, 2h backstop) and the inbound TwiML flow has no pending_call row at all, so keying the alert off pending_call introduces a new lost-alert failure mode; claiming on call_audit is faithful to the table the current dedup already uses.

## root_cause

POST /:id/cancel/refund derives its money-out target from mutable state that its own refunds mutate. ## root_cause
refundExecute's crash-safety design writes the 'pending' proposal_refunds row before calling Stripe so a post-Stripe crash is webhook-adoptable, but the design has no reconciler for the complementary failure quadrants: (1) a crash BEFORE Stripe success leaves a pending row that no charge.refunded event will ever adopt (no refund exists at Stripe), and since e97dfec deliberately makes pending rows block refund headroom, the strand permanently under-refunds and is filtered out of the admin history view, making it undiagnosable without SQL; (2) the error path collapses "Stripe definitively rejected this" and "we don't know if Stripe processed this" into one unconditional status='failed' write (refundExecute.js:68), and a falsely-failed row exits the headroom net, enabling a double-refund on retry. Both quadrants exist because nothing in the system ever consults Stripe as the source of truth for unresolved rows — there is no stripe.refunds.list call anywhere in the codebase.

## root_cause
Non-atomic check-then-act (TOCTOU) on the dead-leg alert dedup. The dedup marker (the call_audit row) is deliberately written only AFTER a successful Telegram send (to preserve retry-on-failed-send), which places the awaited `sendTelegramMessage` HTTP call directly between the existence-check and the row-insert. Two Twilio status callbacks for the same (CallSid, dead-status) — at-least-once redelivery and/or concurrent leg callbacks — interleave inside that await window: both see no row, both send, both insert. Single prod instance does not help because Node's event loop still yields at every await; the network round-trip to the Telegram Bot API is a large, reliable interleaving window. Result: Zul gets a duplicate "that call didn't connect" Telegram alert (and a duplicate audit row).

## fix_design
Make the "claim the right to alert" step atomic by moving the durable write BEFORE the send and using an INSERT ... ON CONFLICT DO NOTHING claim, while preserving the existing retry-on-failed-send property via a release-on-failure DELETE. Three source edits + tests.

1) server/db/schema.sql (SENSITIVE, idempotent) — add a PARTIAL unique index inside the VA-calling block, immediately after `idx_call_audit_created_at` (~line 3513) so `schema.vaCalling.test.js`'s `vaCallingDdl()` slice (from the `-- Zul VA Calling` marker at line 3467 to EOF) picks it up:
   CREATE UNIQUE INDEX IF NOT EXISTS uq_call_audit_dead_leg
     ON call_audit (call_sid, status)
     WHERE call_sid IS NOT NULL
       AND status IN ('no-answer','busy','failed','canceled');
   PARTIAL (not a full UNIQUE) on purpose: call_audit also holds callSid=NULL rows (`rejected_cap`/`rejected_validation` and the placement-`failed` at telegram.js:168) plus one `placed` row per call; scoping to non-null call_sid + the four dead statuses constrains ONLY the dead-leg callback rows we dedup, so it cannot silently DO-NOTHING a spend-cap-relevant insert. (call_sid,'placed') and (call_sid,'failed') are distinct keys, so a call getting both a placed and a dead-leg row is unaffected.

2) server/utils/pendingCall.js — add two helpers and export them:
   claimDeadLegAudit({ triggeredBy, callSid, status }) →
     INSERT INTO call_audit (triggered_by, target_e164, call_sid, status)
     VALUES ($1, NULL, $2, $3)
     ON CONFLICT (call_sid, status)
       WHERE call_sid IS NOT NULL AND status IN ('no-answer','busy','failed','canceled')
       DO NOTHING
     RETURNING id;
     return rows.length > 0;  // true = this caller won the claim and owns the alert
   releaseDeadLegAudit({ callSid, status }) →
     DELETE FROM call_audit WHERE call_sid = $1 AND status = $2;
   (The `ON CONFLICT (cols) WHERE <index_predicate>` form is valid partial-index arbiter inference in Postgres.)

3) server/routes/voice.js — rewire the dead-leg branch of `/status` (replace lines ~173-207). Read `allowed` once at the top of the branch; compute `triggeredBy = allowed ? Number(allowed) : null`. Then:
   - Try `claimed = await _deps.claimDeadLegAudit({ triggeredBy, callSid, status })`. ## root_cause
position is the payroll "money seam" and is compared case-insensitively across the codebase, but two DIFFERENT normalizations are in use: the accrual + clawback + four other payroll sites use LOWER(TRIM(position)) (or the JS isBartender helper, which trims), while cancel.js:328 (and autoAssign.js:149, coverBroadcast.js:156-159) use bare LOWER(position). At cancel time the un-trimmed capture in cancel.js builds bartendersByShift, and that list is handed to clawbackTip as explicitBartenderIds, which OVERRIDES clawbackTip's own trimmed matcher (payrollClawback.js:68-82). So a shift_request row whose position is whitespace-padded (e.g. ' Bartender') is: (a) counted as a bartender by accrual/isBartender and thus actually PAID a card-tip share, but (b) EXCLUDED from the cancel-time capture. Concrete failure at cancel of a booked event that has an accrued card-tip line (the path the existing 'clawback accrued' test at cancel.test.js:446 exercises): if the ONLY approved bartender has a padded position, bartendersByShift.get(shiftId) is [], clawbackTip gets an empty explicit list, resolves 0 users, and takes the allBartenders.length===0 branch — it advances tips.refunded_amount_cents to the full amount WITHOUT writing any negative payout adjustment. The bartender keeps the whole tip AND, because the marker is now burned, a later charge.refunded webhook computes delta<=0 and never claws either — permanent under-collection. In a mixed roster (one clean 'Bartender' + one padded ' Bartender'), the full clawback is split across only the captured (clean) bartender, over-clawing them for the padded bartender's share and never touching the padded one — misdistribution. Whether padded data exists: every ACTIVE write path already canonicalizes — shifts.approval.js:208/416 assign+approve via canonicalizeRole (trims), autoAssign.js:337 hardcodes 'Bartender', staffShiftActions.js:588 cover-claim writes canonicalizeRole(claimerPosition); contractor_profiles.position is not written by any active route (admin/users.js profile PUT and staffPortal PROFILE_ALLOWED_KEYS both omit position). So current routes cannot MINT padded shift_requests.position — the bug is LATENT under normal operation but LIVE for legacy rows predating canonicalization, direct DB edits, or imported data. This is exactly why the P3 campaign trims defensively everywhere; cancel.js was simply the missed site.
