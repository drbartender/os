This is cross-plan ground truth for the three comms SMS plans, written so they can be
authored in parallel and executed without drift. It contains: (a) the scheduled-message
system as it actually exists today (reconciled from a 4-agent recon of `main`, not the
spec's frozen assumptions), (b) **pinned** interfaces for the shared SMS rails and the
priority ladder, and (c) the cross-plan seams.

**The design source of truth is the spec:** `docs/superpowers/specs/2026-05-20-automated-communication-design.md`.
This contract is current-code reality plus interface pins. Where this contract and the
spec's section 10 phasing disagree, **this contract wins** (the spec froze before Phases
0/1/2 shipped). Where this contract and `.plan-2-contract.md` disagree, this contract wins
(see §7).

Read this before writing or executing any of the three plans. Anything marked **PINNED**
is a fixed decision — do not redesign it; other plans depend on it.

## 2. The scheduled-message system today (reconciled ground truth)

Built in Plan 2a. Bootstrapped in `server/index.js`: `preEventHandlers.registerAll()` then
`marketingHandlers.registerMarketingHandlers()` run at boot; `dispatchPending` runs every
300s via `wrapScheduler('message_dispatcher', 300, dispatchPending)`, gated by
`RUN_MESSAGE_DISPATCHER_SCHEDULER`.

### 2.1 Public API

`server/utils/scheduledMessageDispatcher.js`:
- **`registerHandler(messageType, handlerFn, options = {})`** — `options`: `offsetFromEventDate`
  (number of seconds, negative = before anchor, or `null` = anchor-independent / never
  re-anchored); `anchor` (default `'event_date'`); `category` (default `'operational'`).
  `anchor` must be one of `event_date | balance_due_date | created_at | completed_at`
  and `category` one of `operational | marketing` — invalid values throw. Unknown extra
  option keys are currently ignored (relevant to the §5 `priority` seam).
- **`getHandlerMeta(messageType)`** → `{ offsetFromEventDate, anchor, category }` or `null`.
- **`dispatchPending()`** — the scheduler tick. Module-level re-entrancy guard (overlapping
  tick is a no-op). Drains `status='pending' AND scheduled_for <= NOW()` in passes of 100,
  max 50 passes, dispatched sequentially.
- **Handler signature:** `async ({ entity, recipient, scheduledMessage }) => void`.
  Normal return → row marked `'sent'`, `sent_at = NOW()`. **Throwing → row marked
  `'failed'`** with `error_message`. `scheduledMessage` carries only
  `{ id, entity_id, entity_type, message_type, recipient_type, recipient_id, channel,
  scheduled_for }` — no `status`, no `created_at`.

`server/utils/messageScheduling.js`:
- **`scheduleMessage({ entityType, entityId, messageType, recipientType, recipientId, channel, scheduledFor })`**
  — inserts one pending `scheduled_messages` row. Idempotent:
  `ON CONFLICT (entity_id, entity_type, message_type, recipient_id, recipient_type, channel)
  WHERE status = 'pending' DO NOTHING`. **Returns the inserted row `{id, status}`, or
  `null` when a pending duplicate already exists.** `entityId`/`recipientId` must be integers.
  `entityType ∈ proposal|shift|client|consult`, `recipientType ∈ client|staff|admin`,
  `channel ∈ email|sms`.

`server/utils/messageSuppression.js`:
- **`shouldSendImmediate({ proposal, client, channel })`** → `{ ok: true }` or
  `{ ok: false, reason: 'archived' | 'channel_disabled' | 'bad_contact' }`. Pure (no I/O —
  caller passes loaded rows). `client` must carry `.communication_preferences`,
  `.email_status`, `.phone_status`. Used by immediate (non-scheduled) sends. Already
  handles `channel: 'sms'` (checks `sms_enabled` and `phone_status`).

`server/utils/preEventScheduling.js`:
- **`computeScheduledFor(messageType, proposal)`** — applies the handler's offset to the
  anchor date, then forces time-of-day to **10:00 local** in the event timezone
  (`SEND_HOUR_LOCAL = 10`). Throws if the message type is unregistered or has a `null`
  offset. **Do not use this for touches anchored on a wall-clock time** (event-eve,
  day-before staff reminder) — see §8.
- **`schedulePreEventReminders(proposalId, executor)`** — inserts the pre-event ladder rows;
  optional `executor` accepts an in-transaction pg client.

`server/utils/rescheduleProposal.js`:
- **`reanchorPendingMessages(client, proposalId)`** — on reschedule, recomputes
  `scheduled_for` for every pending row using `getHandlerMeta`. **A new scheduled
  message_type with a non-null offset MUST register with correct `offsetFromEventDate`
  and `anchor` or the reschedule cascade silently skips it.** Drip-style touches register
  `offsetFromEventDate: null` so the cascade leaves them alone.

### 2.3 `scheduled_messages` table

Columns: `id, entity_id, entity_type, message_type, recipient_type, recipient_id, channel,
scheduled_for, sent_at, status, error_message, created_at`. `status ∈ pending | sent |
failed | suppressed | deferred`. `channel` already accepts `'sms'`. The `'deferred'` status
is defined and treated as "occupied" by the scheduling helpers but **nothing ever writes
it** — Phase 4b's overlap rule is the first writer.

### 2.4 Anchors and the `completed_at` trap

`proposals` has **no `completed_at` column** — completion is `status = 'completed'`.
A handler registered with `anchor: 'completed_at'` resolves to `undefined` and is silently
skipped by the cascade / throws in `computeScheduledFor`. **Any completion-anchored touch
must anchor on `event_date`** (this is why `review_request` and `retention_nudge` ship
anchored on `event_date`).

### 2.5 Schedulers (all in `server/index.js` `start()`, each `RUN_*_SCHEDULER`-gated)

- `message_dispatcher` — `dispatchPending`, every 5 min.
- `autoAssignScheduler.processScheduledAutoAssigns` — hourly.
- `balanceScheduler.processEventCompletions` — hourly; flips proposals to `completed` when
  `event_date + start_time + duration` is past and balance ≤ 0; per-completion calls
  `scheduleReviewRequest` + `scheduleRetentionNudge`. Natural anchor point for the staff
  post-event thank-you (Phase 4a).

## 3. SMS in the codebase today

- **`server/utils/sms.js`** — `sendSMS({ to, body })` (bare Twilio send; returns the Twilio
  message object, `.sid` is the MessageSid; dev with no Twilio creds returns
  `{ sid: 'dev-skipped' }`). Also `normalizePhone(phone)` (E.164, `null` on unparseable).
  `sendSMS` does **not** log to `sms_messages` and does **not** check preferences.
- **`sms_messages` columns** (from existing INSERTs): `direction` (`inbound|outbound`),
  `client_id`, `sender_id`, `recipient_phone`, `recipient_name`, `body`, `message_type`,
  `status`, `error_message`, `twilio_sid`, `read_at`, `metadata`, `created_at`. Partial
  unique index on `twilio_sid WHERE twilio_sid IS NOT NULL`. **`message_type` is
  `VARCHAR(20)` with `CHECK (message_type IN ('general','invitation','reminder',
  'announcement'))` — too narrow for the Phase 3/4 message_type values. Phase 3's rails
  task widens it to `TEXT` and drops the CHECK as its first migration step (§4).**
- **Phase 2 inbound** is fully built: `POST /api/sms/inbound`, STOP/START
  (flips `communication_preferences.sms_enabled` on `clients` or `users`), CONFIRM/CANT
  (acts on the staffer's nearest approved shift).
- **Admin Messages page** `/messages` joins `sms_messages` by `client_id` — it is
  **client-only**; staff SMS rows (`client_id = null`) never appear there.
- **Nothing schedules SMS through the dispatcher today.** Every SMS send is immediate /
  synchronous. The scheduled SMS path (schema, `scheduleMessage`, `checkSuppression`'s
  `sms` branch) is plumbed but unexercised. Phase 3 is the first to exercise it.

## 4. PINNED — the SMS rails (Phase 3 builds; 4a and 4b consume)

Phase 3 builds these as its opening tasks. Phase 4a and 4b consume them as-is.

1. **`sendAndLogSms(...)` — added to `server/utils/sms.js`.**
   Signature: `async sendAndLogSms({ to, body, clientId = null, messageType, recipientName = null }) => { sid, status }`.
   Behavior: normalize `to` via `normalizePhone` (if `null`, record nothing and return a
   skipped result); call `sendSMS`; INSERT an outbound `sms_messages` row
   (`direction='outbound'`, `client_id`, `recipient_phone`, `recipient_name`, `body`,
   `message_type`, `twilio_sid`, `status`). On Twilio failure: insert the row with
   `status='failed'` + `error_message`, then **throw** (so a scheduled handler's row goes
   `failed` and immediate callers can catch it). This is the single send+log primitive for
   ALL automated SMS in Phases 3/4a/4b. Existing manual SMS paths (`routes/messages.js`,
   `routes/sms.js` reply) are not refactored onto it. The rails task's FIRST step is the
   `sms_messages.message_type` widening migration (§3) — it must land before any
   `sendAndLogSms` call.

2. **`server/utils/smsTemplates.js` — new.** One exported function per SMS touch, returning
   a plain string (the SMS body). Mirrors `emailTemplates.js`. Phase 3 creates it with the
   client SMS copy; Phase 4a appends staff SMS copy. Pull copy verbatim from the spec's
   section 5 templates; render `[placeholders]`; **no em dashes** in body strings.

3. **Scheduled SMS touch pattern.** A scheduled SMS touch = a new `message_type` + a handler
   registered via `registerHandler` + the handler renders via `smsTemplates` and sends via
   `sendAndLogSms`. **You cannot reuse an email handler for an `sms`-channel row** — the
   handler, not the row, decides the send mechanism. Place SMS handlers beside their email
   siblings (drip SMS in `marketingHandlers.js`, etc.); a new handler file is fine if a
   target file is near the size cap.

4. **`message_type` naming — PINNED:**
   - SMS variant of an existing email touch: `<email_type>_sms`
     (`balance_due_today_sms`, `balance_late_t1_sms`, `balance_late_t3_sms`).
   - Drip SMS touches: `drip_touch_1`, `drip_touch_3` (SMS-only, no suffix),
     `drip_touch_5_sms` (the +21d touch has both channels; its email row is
     `drip_touch_5_email`).
   - SMS-only new touch: `event_eve`.
   - Email+SMS new touch (drink-plan nudge): `drink_plan_nudge` (email) +
     `drink_plan_nudge_sms` (SMS).
   - Staff touches (Phase 4a): `shift_reminder` (day-before), `staff_thank_you` (post-event).

5. **Immediate SMS sends** (initial proposal, sign+pay, payment-failure, reschedule) are
   **not** scheduled rows and **not** handlers. They are an SMS send added next to the
   existing email send, gated by `shouldSendImmediate({ proposal, client, channel: 'sms' })`,
   sent via `sendAndLogSms`. Wrap best-effort (own try/catch + Sentry, never rethrown into
   the request path), matching the existing email hooks.

## 5. PINNED — notification priority ladder

Phase 4b builds the overlap-prevention rule (spec 7.4: max 1 message per channel per client
per day, lower-priority loser deferred 24h). It reads a **`priority`** integer from handler
metadata. To keep all plans consistent:

- **`registerHandler` options gain a `priority` field** (integer 1–5, 1 = highest). Phase 4b
  adds it to the options handling and `getHandlerMeta`. Phase 3 and 4a register every new
  SMS handler **with** a `priority` per the table below; the value is inert until Phase 4b
  activates it (unknown option keys are ignored today — see §2.1).
- **Phase 4b also retrofits `priority`** onto the 14 existing email handler registrations,
  per the table below.

| Tier | priority | message types |
|---|---|---|
| Time-sensitive operational | **1** | `event_eve`, `balance_reminder_autopay_t3`, `balance_reminder_non_autopay_t3`, `balance_due_today`(+`_sms`) |
| Action-required | **2** | `balance_late_t1`(+`_sms`), `balance_late_t3`(+`_sms`), `drink_plan_nudge`(+`_sms`) |
| Lifecycle | **3** | `event_week_reminder`, `long_lead_t30_recap`, `review_request` |
| Drip | **4** | `drip_touch_1`, `drip_touch_2`, `drip_touch_3`, `drip_touch_4`, `drip_touch_5_email`, `drip_touch_5_sms` |
| Marketing / retention | **5** | `new_year_hello`, `six_months_out`, `retention_nudge` |

**Hard cooldown exceptions** (immune from the 24h deferral — must fire on their exact day):
`event_eve` and `balance_due_today`(+`_sms`). Phase 4b implements the exception; Phase 3
flags these two handlers (e.g. a `cooldownExempt: true` option Phase 4b also adds).

Staff touches (`shift_reminder`, `staff_thank_you`) are priority 1 and 3 respectively but
the per-client/day cooldown is a **client** rule — Phase 4b decides whether staff get an
equivalent per-staff cap; staff volume is low, so a staff cap is optional.

**Multi-channel touches — no channel substitution.** Per spec 7.3, a touch scheduled on
BOTH email and SMS (two independent rows) must not have either row channel-substituted by
Phase 4b's delivery-failure fallback — substituting one would duplicate the other.
`registerHandler` gains a third option, `multiChannel` (boolean, default false); Phase 4b's
`resolveDelivery` suppresses (does not substitute) a `multiChannel` row whose own channel is
dead, letting the paired row carry delivery. The multi-channel set is 5 pairs:
`drink_plan_nudge`/`drink_plan_nudge_sms`, `balance_due_today`/`balance_due_today_sms`,
`balance_late_t1`/`balance_late_t1_sms`, `balance_late_t3`/`balance_late_t3_sms`,
`drip_touch_5_email`/`drip_touch_5_sms`. Phase 3 sets `multiChannel: true` on the 6 new
handlers it creates; Phase 4b sets it on the 4 existing email handlers during its retrofit.

## 6. Cross-plan seams

- **`server/utils/scheduledMessageDispatcher.js` is edited by 4a and 4b, not by Phase 3.**
  - Phase 3: registers new SMS handlers and adds scheduling calls only. It does **not**
    modify the dispatcher core — `checkSuppression` already handles `sms`-channel `client`
    rows.
  - Phase 4a: adds `recipient_type IN ('staff','admin')` branches to `checkSuppression`
    (staff/admin comm-prefs; `users.communication_preferences` already exists and is honored
    by STOP-keyword).
  - Phase 4b: adds `priority`, `cooldownExempt`, and `multiChannel` to
    `registerHandler`/`getHandlerMeta`; adds the overlap-prevention pass to the dispatch
    loop (priority-ordered, per-batch in-memory sort); adds a `resolveDelivery` step that
    takes over per-channel client suppression and channel substitution from
    `checkSuppression`; retrofits `priority` (and `multiChannel` where applicable) onto the
    14 existing registrations.
  - 4a and 4b touch different functions of this file — keep their edits in separate commits;
    coordinate if both are in flight.
- **Drip-suppression query.** `marketingHandlers.js` cancels pending drip rows when a
  proposal signs+pays (the query lists drip `message_type`s explicitly, ~line 496-505).
  **Phase 3 owns updating that list** to include `drip_touch_1`, `drip_touch_3`,
  `drip_touch_5_sms`, or signed clients keep getting drip SMS.
- **`schedulePreEventReminders`** (`preEventScheduling.js`) — Phase 3 extends it to also
  schedule `drink_plan_nudge` rows. `event_eve` needs bespoke timing (§8) — Phase 3 decides
  whether it rides this helper or a separate one.
- **`_deps` test seam.** `lifecycle.js` and `crud.js` use a `_deps` injection object so
  best-effort hooks are testable. New immediate-SMS hooks added there should follow it.
- **Reschedule cascade.** New non-null-offset scheduled SMS types must register correct
  `offsetFromEventDate`/`anchor` (§2.1).

## 8. Carried-forward gaps & gotchas

- **`event_eve` timing.** `scheduled_for` is the event start timestamp minus 24h, in event
  TZ. It does **not** use the 10:00-local convention of `computeScheduledFor`. Build bespoke
  timing. Same for Phase 4a's `shift_reminder` (T-24h before event start).
- **`drink_plan_nudge` is a full build.** The spec routed it to Phase 1 but it was never
  built — no email side, no template, no scheduling. Phase 3 builds the whole touch (email +
  SMS, per spec 3.7), scheduled T-21 from event, suppressed when
  `drink_plans.selections IS NOT NULL OR drink_plans.consult_filled_at IS NOT NULL`.
- **`users` has no `phone` column.** Staff/admin phone lives on `contractor_profiles.phone`.
  Phase 4b's multi-admin fan-out must `JOIN contractor_profiles`; an admin may have no
  `contractor_profiles` row at all (→ email-only for that admin). The spec's literal
  `SELECT id, phone, email FROM users` query is wrong.
- **Sign+pay coverage is non-uniform.** Three paths reach sign+pay: `payment_intent.succeeded`
  (full hooks), `checkout.session.completed` Payment-Link (missing `schedulePreEventReminders`
  / `scheduleBalanceReminders` — a known follow-up, out of scope here), admin record-payment.
  Phase 3's sign+pay SMS rides wherever the orientation email rides (the `isCoupledSigning`
  branch of `sendPaymentNotifications`), inheriting the same coverage — do not try to fix
  the webhook gap.
- **Legacy drip coexistence.** A legacy "Abandoned Quote Followup" `email_sequence`
  campaign may still run alongside the new drip. Phase 3 should verify it is paused before
  adding drip SMS (else double-touch).
- **`users.notification_preferences`** exists in schema (11 categories) but is read by zero
  code — Phase 4b builds the first reader.
- **Resend bounce webhook.** Phase 4b's delivery-fallback needs a writer for
  `clients.email_status='bad'`. Investigate whether a Resend inbound/bounce webhook +
  `email_webhook_events` table exist; if absent, that webhook is part of 4b's net-new work.

### Phase 4b — Notification Infrastructure

Three task groups, each independently committable:

1. **Overlap prevention (spec 7.4):** add `priority` + `cooldownExempt` to
   `registerHandler`/`getHandlerMeta`; retrofit `priority` onto the 14 existing handlers
   per §5; implement the dispatch-time rule (max 1 per channel per client per day, lower
   priority loses and is deferred 24h via `status='deferred'`, hard exceptions for
   `event_eve` / `balance_due_today`).
2. **Delivery-failure fallback (spec 7.5 + 7.3):** writers that flip
   `clients.email_status` / `phone_status` to `'bad'` on a Resend hard bounce / Twilio
   delivery failure (investigate the existing Resend webhook — see §8); channel substitution
   for scheduled single-channel operational touches; "both bad" → suspend the client's
   automation + admin alert.
3. **Multi-admin notification model (spec 8.3 + 6):** read `users.notification_preferences`;
   build a central admin-notification helper with the category fan-out query (`JOIN
   contractor_profiles` for phone — §8); a per-category toggle UI in user settings; migrate
   the ~13 scattered `ADMIN_EMAIL`/`ADMIN_PHONE` call sites onto it.
